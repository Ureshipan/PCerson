# Техническое задание

## Проект: локальный desktop/home AI-ассистент

## 1. Назначение проекта

Разработать локального AI-ассистента, который в базовом варианте живёт на ПК пользователя и работает как “личность внутри компьютера”, а в расширенном варианте может управлять устройствами умного дома и дополнительной периферией.

Система должна быть:

* локальной по умолчанию;
* воспроизводимой после переустановки системы;
* конфигурируемой без правки исходного кода или с минимальной правкой;
* модульной;
* пригодной для Windows в первую очередь;
* расширяемой под Linux на следующем этапе;
* устойчивой к отсутствию отдельных устройств и сервисов.

---

## 2. Главные цели

### 2.1. Базовые цели v1

Реализовать ассистента, который:

* запускается локально на Windows-PC;
* использует локальную LLM;
* поддерживает долгосрочную память о пользователе;
* умеет реагировать текстом, а не голосом;
* может открывать программы, файлы, папки и ярлыки;
* может использовать микрофон и камеру;
* может использовать термо-принтер, если он подключён;
* поддерживает несколько способов активации;
* допускает наличие или отсутствие Home Assistant без поломки базовой работы;
* управляется централизованным конфигом;
* может быть развёрнут заново с минимальным человеческим фактором.

### 2.2. Цели v1.5 / v2

* интеграция с Home Assistant;
* подключение и управление smart-home устройствами;
* автономное поведение на основе пользовательских действий;
* Linux-адаптация;
* дополнительные устройства: LED-лента, датчики, ESP32 и т.д.;
* более развитая персонализация персонажа и стиля.

---

## 3. Не-цели

На этапе v1 проект не должен требовать:

* обязательного облачного API;
* обязательного TTS;
* обязательного Home Assistant;
* полной удалённой автоматизации всего компьютера;
* “полного контроля над системой” через AI;
* multi-user поддержки;
* публичного сетевого доступа из интернета;
* сложной распределённой архитектуры.

---

## 4. Основные принципы

1. **Local-first**
   Всё критичное должно работать без интернета.

2. **Desktop-first**
   Базовая среда обитания ассистента — ПК пользователя.

3. **Home Assistant optional**
   Home Assistant — опциональный device hub, а не обязательное ядро.

4. **Config-driven**
   Подключение устройств, способов активации, моделей, личности и поведения должно задаваться конфигом.

5. **Graceful degradation**
   Если нет камеры, принтера, ламп, Home Assistant или сети — система не должна падать, только терять часть функций.

6. **Reproducibility**
   После переустановки системы проект должен подниматься из репозитория и пользовательских конфигов.

7. **Separation of concerns**
   Личность, память, интерфейсы устройств, запуск программ, модельный runtime и интеграция с умным домом должны быть разделены по модулям.

8. **Windows-first, Linux-ready**
   Архитектура сразу должна предусматривать платформенный слой.

## 6. Общая архитектура

Система состоит из 4 логических слоёв:

### 6.1. `assistant-core`

Главное приложение. Отвечает за:

* диалоговую логику;
* orchestration;
* память;
* личность;
* выбор реакции;
* capability routing;
* правила инициативности;
* работу с провайдерами данных;
* интеграции с внешними сервисами.

### 6.2. `host-agent`

Локальный агент, работающий на хостовой ОС. Отвечает за:

* микрофон;
* камеру;
* клавиатурные hotkey;
* desktop notifications / окна;
* запуск программ и файлов;
* Bluetooth-принтер;
* доступ к системным возможностям Windows.

### 6.3. `service layer`

Внешние, но локальные сервисы:

* LLM runtime;
* STT runtime;
* при необходимости TTS runtime;
* MQTT broker;
* опционально Home Assistant.

### 6.4. `config + state layer`

* пользовательские конфиги;
* локальная БД памяти;
* кэш провайдеров;
* runtime state;
* логи.

## 8. Функциональные требования

# 8.1. Режимы активации

Система должна поддерживать несколько способов активации, каждый из которых может включаться/отключаться через конфиг.

### Обязательные режимы

1. **Hotkey activation**

   * глобальная горячая клавиша;
   * настраивается;
   * может быть включена/отключена.

2. **Wake word activation**

   * через микрофон;
   * настраиваемое ключевое слово;
   * можно отключить полностью;
   * можно выбрать устройство ввода.

3. **Manual / text invocation**

   * через CLI-команду;
   * через внутренний debug/dev режим.

### Отложенный режим

4. **Autonomous activation**

   * ассистент сам инициирует действие;
   * основан на активности пользователя или системном контексте;
   * должен быть предусмотрен архитектурно, но не обязателен для реализации в первой версии.

### Требования к конфигу активации

* можно включить один, несколько или все режимы;
* для каждого режима задаются свои параметры;
* дефолтный конфиг генерируется автоматически.

---

# 8.2. Работа с устройствами ввода/вывода

Система должна уметь выбирать конкретные устройства при наличии нескольких.

### Требования

* выбор микрофона по имени/идентификатору;
* выбор камеры по имени/идентификатору;
* выбор аудиоустройства при необходимости;
* выбор принтера по конфигу;
* в случае отсутствия выбранного устройства — корректный fallback и предупреждение в логах.

### Ожидаемое поведение

* при первом запуске возможно автообнаружение устройств;
* дефолтный конфиг должен быть сгенерирован на основании обнаруженного окружения;
* пользователь может затем переопределить выбор вручную;
* пользовательские настройки не должны перезаписываться при обновлении.

---

# 8.3. Реакции ассистента

Основной канал реакции в v1 — **текст**, а не голос.

### Обязательные формы реакции

1. **Текстовые desktop уведомления**
2. **Отдельные окна/overlay с сообщением**
3. **Консольные окна / terminal-style messages**
4. **Запись текстового файла / log note**
5. **Печать на термо-принтере** при наличии устройства

### Требования

* способ реакции выбирается политикой и конфигом;
* разные типы реакции могут комбинироваться;
* сообщения не должны мешать работе пользователя сильнее, чем настроено;
* должна быть возможность задать “тихий режим”;
* должна быть возможность полностью отключить активные всплывающие окна.

### Отложенные реакции

* TTS;
* анимированные UI-аватары;
* сложные визуальные темы.

---

# 8.4. Desktop automation

Ассистент должен уметь выполнять ограниченный набор безопасных действий на компьютере.

### Обязательные действия

* открыть программу;
* открыть ярлык;
* открыть файл;
* открыть папку;
* открыть URL;
* открыть недавний файл;
* открыть недавнюю папку;
* запустить заранее зарегистрированный пользовательский скрипт или команду.

### Ограничения

* ассистент не должен иметь “безграничный shell-доступ” по умолчанию;
* выполнение команд должно идти через whitelist / capability layer;
* доступ к путям должен быть ограничен политикой;
* потенциально опасные действия должны быть вынесены в отдельный capability с явным разрешением.

### User-friendly требования

* пользователь может объявить список приложений и ярлыков в конфиге;
* допускается режим автоиндексации рабочего стола, Start Menu и частых путей;
* недавние файлы/папки должны подбираться безопасным и прозрачным способом.

---

# 8.5. Память

Ассистент должен поддерживать долговременную память.

### Типы памяти

1. **Profile memory**

   * факты о пользователе;
   * предпочтения;
   * ограничения;
   * известные привычки.

2. **Session memory**

   * контекст текущего взаимодействия.

3. **Episodic memory**

   * краткие summaries прошлых взаимодействий.

4. **System memory**

   * известные устройства;
   * доступные capability;
   * ранее успешные способы действий.

### Требования

* память должна храниться локально;
* быть переносимой;
* поддаваться бэкапу;
* иметь политику retention;
* иметь механизм суммаризации;
* не зависеть от облачных сервисов.

### Рекомендуемый подход

* SQLite как основная БД v1;
* абстракция memory service, чтобы позже можно было заменить внутреннюю реализацию.

---

# 8.6. Персона и стиль

Система должна поддерживать настраиваемую личность ассистента.

### Требования

* личность описывается конфигом;
* можно менять tone/style без правки кода;
* можно задавать:

  * стиль речи;
  * уровень иронии;
  * интенсивность “вайба”;
  * словарь;
  * допустимые/недопустимые формулировки;
  * ограничения на инициативность;
  * шаблоны системных сообщений.

### Важно

* дефолтная персона должна быть нейтральной;
* пользовательская персона не должна коммититься в git по умолчанию;
* должен существовать пример конфигурации персонажа.

---

# 8.7. Камера

Камера — опциональная capability.

### v1

* выбрать активную камеру;
* сделать снимок по запросу;
* передать изображение vision-модели;
* использовать камеру только при разрешении в конфиге.

### Требования

* отключаемость;
* логирование обращений к камере;
* возможность выбрать несколько камер в будущем;
* отсутствие камеры не ломает систему.

---

# 8.8. Микрофон / STT

STT — поддерживаемый, но не обязательный путь ввода.

### Требования

* локальное распознавание речи;
* выбор конкретного микрофона;
* возможность полного отключения voice input;
* возможность использовать только hotkey и текстовые команды.

### Важно

* если STT отключён, остальная система должна полноценно работать.

---

# 8.9. Принтер

Термо-принтер — опциональная capability.

### Требования

* включаемый/отключаемый модуль;
* работа через Python backend;
* возможность печатать:

  * текст;
  * простую форматированную заметку;
  * изображение, если backend поддерживает;
* при отсутствии или выключении принтера сценарий должен деградировать без падения системы.

### Важно

* состояние “принтер выключен физически” должно обрабатываться корректно;
* ошибку печати нужно логировать;
* реакция может быть заменена на popup или log note.

---

# 8.10. Home Assistant integration

Home Assistant должен быть опциональным модулем.

### Если модуль включён

Ассистент может:

* обнаруживать сущности;
* выполнять команды к поддерживаемым устройствам;
* читать состояние устройств;
* использовать smart-home как capability layer.

### Если модуль выключен

Ассистент остаётся полностью работоспособным в desktop-режиме.

### Требования

* интеграция через официальные API/протоколы;
* настройка через конфиг;
* список доступных устройств не хардкодится в исходниках;
* user-friendly включение/отключение.

---

# 8.11. Провайдеры внешних данных

Погодные и новостные провайдеры допустимы как внешние зависимости.

### Требования

* только через provider abstraction layer;
* несколько провайдеров в конфиге с приоритетом;
* fallback между провайдерами;
* кэширование;
* отсутствие провайдера не ломает ассистента.

## 9. Нефункциональные требования

# 9.1. Воспроизводимость

Проект должен быть поднимаемым:

* из git-репозитория;
* из шаблонных конфигов;
* через bootstrap-скрипты;
* с минимальными ручными действиями.

# 9.2. Версионирование

* весь код и шаблонные конфиги — в git;
* пользовательские секреты и персональные конфиги — вне git;

# 9.3. Надёжность

* отказ отдельных устройств/модулей не валит всё приложение;

# 9.4. Безопасность

* минимум опасных системных действий по умолчанию;
* явное разграничение capability;
* whitelist для разрешённых действий;
* пользователь может отключить чувствительные функции.

# 9.5. Переносимость конфигов

* пользовательские настройки должны быть централизованы;
* перенос на другую систему должен требовать копирования ограниченного набора каталогов/файлов;
* шаблоны конфигов генерируются автоматически.

## 10. Конфигурационная модель

## 10.1. Общие требования к конфигу

* конфиг централизован;
* поддерживается автоматическая генерация дефолтных конфигов;
* пользовательские конфиги не коммитятся в git по умолчанию;
* примерные конфиги и schema-файлы коммитятся;
* возможна валидация конфигов;
* возможен merge default + user override.

## 10.2. Рекомендуемая схема каталогов

```text
config/
  defaults/
    assistant.default.yaml
    devices.default.yaml
    persona.default.yaml
    models.default.yaml
    providers.default.yaml
    routines.default.yaml
  user/
    assistant.yaml
    devices.yaml
    persona.yaml
    models.yaml
    providers.yaml
    routines.yaml
```

### Правила

* `defaults/` хранятся в git;
* `user/` игнорируются git;
* при первом запуске, если `user/*.yaml` отсутствуют, они генерируются автоматически из дефолтных шаблонов и, при необходимости, из результатов автообнаружения устройств.

## 10.3. Что должно быть в конфиге

### `assistant.yaml`

* включённые capability;
* включённые методы активации;
* общие настройки поведения;
* режимы уведомлений;
* платформа;
* логирование.

### `devices.yaml`

* микрофоны;
* камеры;
* принтеры;
* desktop aliases;
* smart-home entities;
* доступные устройства и их backends.

### `persona.yaml`

* стиль;
* тон;
* ограничения;
* интенсивность ролевого вайба;
* шаблоны сообщений.

### `models.yaml`

* LLM backend;
* STT backend;
* vision backend;
* модель по умолчанию;
* fallback модели;
* пути/названия.

### `providers.yaml`

* weather providers;
* news providers;
* таймауты;
* fallback;
* кэш.

### `routines.yaml`

* автономное поведение;
* quiet hours;
* инициативность;
* политика выбора канала реакции.

## 11. Git policy

В git должны попадать:

* исходный код;
* документация;
* примеры конфигов;
* схема конфигов;
* bootstrap/doctor scripts;
* Docker/Compose конфигурация;
* тесты.

В git не должны попадать:

* пользовательские конфиги;
* secrets;
* базы памяти;
* кэши;
* runtime state;
* локальные логи;
* модельные веса, если это не осознанное решение.

Должен быть `.gitignore`, учитывающий:

* `config/user/`
* `state/`
* `.env`
* локальные артефакты модели/кэша

## 12. Требования к развертыванию

## 12.1. Должен быть bootstrap

Нужен автоматизированный способ подготовки окружения на Windows.

### Задачи bootstrap

* проверить наличие Docker;
* проверить наличие Python;
* проверить GPU/драйверы при необходимости;
* подтянуть контейнерные сервисы;
* подтянуть внешние зависимости;
* создать пользовательские конфиги, если их нет;
* предложить или выполнить загрузку моделей;
* выполнить первичную диагностику.

### Формат

* основной bootstrap для Windows: `bootstrap.ps1`
* позже аналог для Linux: `bootstrap.sh`

## 12.2. Должен быть doctor

Команда диагностики должна проверять:

* наличие нужных сервисов;
* доступность моделей;
* корректность конфигов;
* наличие устройств;
* доступность камеры/микрофона/принтера;
* доступность Home Assistant, если включён;
* доступность провайдеров, если включены.

---

## 13. Контейнеризация

### Требования

* сервисная часть должна быть контейнеризована;
* использовать `docker compose`;
* host-level доступ к устройствам не должен насильно загоняться в контейнер, если это мешает стабильности.

### Рекомендация

В контейнерах:

* `assistant-core`
* `ollama` или альтернативный runtime (у пользователя должен быть выбор, какую модель использовать!)
* `mqtt`
* опционально `home-assistant`
* опционально вспомогательные сервисы

Вне контейнеров:

* `host-agent`
* прямой доступ к камере/микрофону/hotkeys/Bluetooth/desktop APIs

---

## 14. Интерфейсы модулей

Нужна модульная plugin/capability архитектура.

### 14.1. Capability adapter

Каждый адаптер должен предоставлять унифицированный интерфейс:

* `name`
* `enabled`
* `healthcheck()`
* `capabilities()`
* `execute(action, payload)`

### 14.2. Provider adapter

Каждый provider должен предоставлять:

* `name`
* `healthcheck()`
* `fetch()`
* `normalize()`

### 14.3. Activation adapter

Каждый способ активации должен предоставлять:

* `start()`
* `stop()`
* `is_enabled()`
* `emit_event()`

### 14.4. Platform adapter

Платформенный слой должен абстрагировать:

* открытие программ;
* открытие файлов;
* поиск ярлыков;
* recent files;
* notifications;
* windows/overlay behavior;
* hotkeys.

* секреты.

Переход на новую машину должен сводиться к:

4. перепроверке устройств.

## 17. Минимальный user flow

### Первый запуск

1. пользователь клонирует репозиторий;
2. запускает bootstrap;
3. генерируются дефолтные пользовательские конфиги;
4. система пытается обнаружить устройства;
5. пользователь правит только `config/user/*.yaml`, если хочет;
6. запускается doctor;
7. ассистент готов к базовой работе.

### Типичный daily use

* активация hotkey или wake word;
* пользователь даёт команду;
* ассистент понимает intent;
* routing на capability;
* действие выполняется;
* ассистент отвечает текстом/окном/печатью;
* при необходимости обновляется память.

## 18. Ограничения и важные решения

1. **TTS не является обязательной частью v1.**
2. **Home Assistant не является обязательным для базовой работы.**
3. **Все smart-home функции должны быть отключаемы.**
4. **Все desktop действия идут через capability/policy слой.**
5. **Никаких обязательных облачных зависимостей.**
6. **Windows — первый приоритет.**
7. **Автономное поведение — закладывается архитектурно, но можно реализовать позже.**

---

## 19. Критерии готовности v1

Версия считается готовой, если:

1. проект поднимается на Windows по инструкциям из репозитория;
2. генерируются дефолтные пользовательские конфиги;
3. пользовательские конфиги не попадают в git;
4. ассистент работает без Home Assistant;
5. ассистент поддерживает hotkey-активацию;
6. ассистент поддерживает хотя бы один текстовый канал реакции;
7. ассистент умеет открывать программы/файлы/папки;
8. ассистент хранит базовую долговременную память;
9. доступен doctor/диагностический режим;
10. принтер, микрофон, камера и smart-home модули могут быть включены/выключены конфигом;
11. отсутствие любого из этих модулей не ломает систему.

## 21. Короткая формулировка для AI-кодера

Нужно спроектировать и реализовать локального модульного AI-ассистента для Windows, живущего в первую очередь внутри компьютера, а не внутри smart-home системы. Ассистент должен иметь собственное ядро, собственную память, централизованные конфиги, безопасные desktop capability, опциональные модули камеры/микрофона/принтера/Home Assistant и удобную воспроизводимую установку через bootstrap. Все пользовательские настройки и состояние должны быть переносимыми и не попадать в git. Базовая работа обязана сохраняться даже при полном отсутствии Home Assistant и устройств умного дома.
